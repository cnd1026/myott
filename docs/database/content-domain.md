# Content Domain Database Design Draft

MyOTT의 콘텐츠 도메인을 위한 DB 설계 초안입니다. 이 문서는 테이블 후보와 주요 컬럼 방향을 정리하기 위한 문서이며, SQL 작성, Supabase 연결, ERD 작성은 포함하지 않습니다.

## 1. Content Domain 개요

Content Domain은 MyOTT가 추천하고 보여주는 작품 정보를 담당합니다. 현재 MVP는 영화, 드라마, 애니를 중심으로 하지만, 이후 만화, 웹툰, 소설, 게임, 음악까지 확장될 수 있어야 합니다.

이 도메인의 핵심 역할은 다음과 같습니다.

- 추천 카드에 표시할 작품 기본 정보 관리
- 상세 Layer에 표시할 줄거리, 포스터, 장르, 감독, 배우 정보 관리
- TMDB 같은 외부 데이터 제공자와 내부 콘텐츠 데이터 연결
- OTT 플랫폼별 시청 가능 여부 관리
- 국가, 언어, 컬렉션 같은 분류 정보 관리
- 사용자 데이터와 콘텐츠 메타데이터를 분리해 개인정보 저장 범위를 줄이는 기반 제공

### 설계 경계

- 이 문서는 DB 설계 초안입니다.
- SQL은 작성하지 않습니다.
- ERD는 그리지 않습니다.
- 실제 Supabase 테이블 생성은 하지 않습니다.
- TMDB API 연결은 하지 않습니다.

## 2. 필요한 테이블 후보

- `contents`
- `genres`
- `content_genres`
- `people`
- `content_people`
- `ott_platforms`
- `content_platforms`
- `countries`
- `languages`
- `collections` (선택)

## 3. 테이블별 설계 초안

### contents

#### 목적

MyOTT에서 추천, 검색, 상세 조회의 중심이 되는 작품 기본 정보를 저장합니다.

#### 주요 컬럼(초안)

- `id`: 내부 콘텐츠 ID
- `external_source`: TMDB 등 외부 데이터 출처
- `external_id`: 외부 서비스의 콘텐츠 ID
- `content_type`: movie, drama, animation 등 콘텐츠 타입
- `title`: 표시 제목
- `original_title`: 원제
- `overview`: 짧은 줄거리
- `poster_path`: 포스터 이미지 경로 또는 URL
- `backdrop_path`: 배경 이미지 경로 또는 URL
- `release_date`: 공개일
- `runtime_minutes`: 러닝타임
- `age_rating`: 관람 등급
- `primary_country_id`: 대표 제작 국가
- `original_language_id`: 원어
- `status`: released, upcoming, ended 등 상태
- `popularity_score`: 외부 또는 내부 인기 점수
- `created_at`: 생성 시각
- `updated_at`: 수정 시각

#### 왜 필요한지

추천 결과 카드와 상세 Layer의 기준이 되는 핵심 테이블입니다. 작품 제목, 포스터, 줄거리, 콘텐츠 타입, 공개일 같은 기본 정보가 한곳에 있어야 추천 결과와 상세 화면을 안정적으로 구성할 수 있습니다.

#### 향후 확장 가능성

- 영화, 드라마, 애니 외에 만화, 웹툰, 소설, 게임, 음악 타입을 추가할 수 있습니다.
- TMDB 외 다른 데이터 제공자를 연결할 수 있습니다.
- AI 추천용 임베딩이나 내부 점수는 별도 테이블로 분리할 수 있습니다.
- 종료 예정, 신규 공개 예정 기능과 연결될 수 있습니다.

### genres

#### 목적

장르 목록을 표준화해 저장합니다.

#### 주요 컬럼(초안)

- `id`: 장르 ID
- `name`: 장르명
- `slug`: 내부 식별용 문자열
- `external_source`: 외부 데이터 출처
- `external_id`: 외부 장르 ID
- `description`: 장르 설명
- `created_at`: 생성 시각
- `updated_at`: 수정 시각

#### 왜 필요한지

Quick Pick과 추천 필터에서 장르를 안정적으로 사용하려면 장르명이 중복되거나 흔들리지 않아야 합니다. SF, 로맨스, 스릴러 같은 장르를 공통 기준으로 관리합니다.

#### 향후 확장 가능성

- TMDB 장르와 MyOTT 내부 장르를 매핑할 수 있습니다.
- 콘텐츠 타입별 장르 분류를 추가할 수 있습니다.
- 사용자 취향 DNA에서 선호 장르 계산에 사용할 수 있습니다.

### content_genres

#### 목적

작품과 장르의 다대다 관계를 저장합니다.

#### 주요 컬럼(초안)

- `id`: 연결 ID
- `content_id`: 작품 ID
- `genre_id`: 장르 ID
- `weight`: 대표 장르 여부 또는 중요도
- `created_at`: 생성 시각

#### 왜 필요한지

하나의 작품은 여러 장르를 가질 수 있고, 하나의 장르는 여러 작품에 연결됩니다. 예를 들어 `인터스텔라`는 SF와 드라마를 동시에 가질 수 있습니다.

#### 향후 확장 가능성

- 장르별 추천 점수 계산에 사용할 수 있습니다.
- 대표 장르와 보조 장르를 구분할 수 있습니다.
- Quick Pick 장르 필터와 직접 연결할 수 있습니다.

### people

#### 목적

감독, 배우, 작가 등 콘텐츠와 관련된 인물 정보를 저장합니다.

#### 주요 컬럼(초안)

- `id`: 인물 ID
- `external_source`: TMDB 등 외부 데이터 출처
- `external_id`: 외부 인물 ID
- `name`: 표시 이름
- `original_name`: 원어 이름
- `profile_path`: 프로필 이미지 경로 또는 URL
- `known_for_department`: acting, directing, writing 등 대표 분야
- `created_at`: 생성 시각
- `updated_at`: 수정 시각

#### 왜 필요한지

상세 Layer에는 감독과 주요 배우가 표시됩니다. 인물 정보를 작품 문자열 안에 직접 저장하면 중복이 많아지고, 같은 배우가 여러 작품에 등장할 때 관리가 어렵습니다.

#### 향후 확장 가능성

- 배우 또는 감독 기반 추천으로 확장할 수 있습니다.
- 인물 상세 페이지를 만들 수 있습니다.
- 출연작 목록, 감독작 목록을 구성할 수 있습니다.

### content_people

#### 목적

작품과 인물의 관계, 역할, 출연 순서를 저장합니다.

#### 주요 컬럼(초안)

- `id`: 연결 ID
- `content_id`: 작품 ID
- `person_id`: 인물 ID
- `role_type`: actor, director, writer 등 역할 타입
- `character_name`: 배우의 배역명
- `credit_order`: 화면 표시 순서
- `created_at`: 생성 시각

#### 왜 필요한지

한 작품에는 여러 배우와 감독이 있고, 한 인물은 여러 작품에 참여합니다. 작품별로 감독과 주요 배우를 구분해 표시하려면 관계 테이블이 필요합니다.

#### 향후 확장 가능성

- 주요 배우만 추천 카드에 노출하는 기준을 만들 수 있습니다.
- 특정 감독 또는 배우를 선호하는 사용자에게 추천할 수 있습니다.
- 작가, 음악감독, 제작자 등 역할을 확장할 수 있습니다.

### ott_platforms

#### 목적

Netflix, Disney+, Watcha, TVING 등 OTT 플랫폼 정보를 저장합니다.

#### 주요 컬럼(초안)

- `id`: OTT 플랫폼 ID
- `name`: 플랫폼명
- `slug`: 내부 식별용 문자열
- `homepage_url`: 공식 웹사이트 URL
- `logo_path`: 로고 이미지 경로 또는 URL
- `country_id`: 대표 서비스 국가
- `is_active`: 현재 사용 여부
- `created_at`: 생성 시각
- `updated_at`: 수정 시각

#### 왜 필요한지

사용자는 자신이 이용 중인 OTT를 선택하고, MyOTT는 해당 OTT에서 볼 수 있는 작품을 우선 추천해야 합니다. 플랫폼 목록을 표준화해 관리해야 필터가 안정적으로 동작합니다.

#### 향후 확장 가능성

- 지역별 OTT 제공 여부를 분리할 수 있습니다.
- OTT 종료 예정, 신규 공개 예정 기능과 연결할 수 있습니다.
- 사용자의 OTT 구독 정보와 연결할 수 있습니다.

### content_platforms

#### 목적

작품이 어떤 OTT 플랫폼에서 제공되는지 저장합니다.

#### 주요 컬럼(초안)

- `id`: 연결 ID
- `content_id`: 작품 ID
- `ott_platform_id`: OTT 플랫폼 ID
- `country_id`: 제공 국가
- `availability_type`: stream, rent, buy 등 제공 방식
- `available_from`: 제공 시작일
- `available_until`: 제공 종료 예정일
- `source`: 제공 정보 출처
- `last_checked_at`: 마지막 확인 시각
- `created_at`: 생성 시각
- `updated_at`: 수정 시각

#### 왜 필요한지

같은 작품이라도 OTT와 국가에 따라 볼 수 있는지 여부가 달라집니다. 추천 결과에서 사용자가 실제로 볼 수 있는 작품을 우선 보여주려면 작품과 플랫폼의 관계가 필요합니다.

#### 향후 확장 가능성

- OTT 종료 예정 알림을 만들 수 있습니다.
- 신규 공개 예정 목록을 만들 수 있습니다.
- 국가별 콘텐츠 제공 상태를 관리할 수 있습니다.
- 무료, 대여, 구매 같은 제공 방식을 구분할 수 있습니다.

### countries

#### 목적

제작 국가, 서비스 국가, 사용자의 지역 필터에 사용할 국가 정보를 저장합니다.

#### 주요 컬럼(초안)

- `id`: 국가 ID
- `name`: 국가명
- `iso_code`: ISO 국가 코드
- `region`: 지역 분류
- `created_at`: 생성 시각
- `updated_at`: 수정 시각

#### 왜 필요한지

Quick Pick의 국가 필터와 OTT 제공 국가, 작품 제작 국가를 같은 기준으로 관리해야 합니다. 국가명이 문자열로 흩어지면 검색과 필터링이 어려워집니다.

#### 향후 확장 가능성

- 국가별 인기 콘텐츠 추천에 사용할 수 있습니다.
- 서비스 지역 확장 시 국가별 설정을 붙일 수 있습니다.
- TMDB 지역 코드와 연결할 수 있습니다.

### languages

#### 목적

작품의 원어, 자막, 더빙, 검색 언어 등을 관리할 언어 정보를 저장합니다.

#### 주요 컬럼(초안)

- `id`: 언어 ID
- `name`: 언어명
- `iso_code`: ISO 언어 코드
- `created_at`: 생성 시각
- `updated_at`: 수정 시각

#### 왜 필요한지

작품 원어와 사용자 언어 설정은 추천과 표시 품질에 영향을 줍니다. TMDB 같은 외부 데이터는 언어 코드 기반으로 동작하므로 내부에서도 언어 기준이 필요합니다.

#### 향후 확장 가능성

- 다국어 제목과 줄거리 테이블로 확장할 수 있습니다.
- 자막/더빙 제공 여부와 연결할 수 있습니다.
- 사용자 언어 설정에 맞는 콘텐츠 표시로 확장할 수 있습니다.

### collections (선택)

#### 목적

시리즈, 프랜차이즈, 묶음 콘텐츠 정보를 저장합니다.

#### 주요 컬럼(초안)

- `id`: 컬렉션 ID
- `external_source`: 외부 데이터 출처
- `external_id`: 외부 컬렉션 ID
- `name`: 컬렉션명
- `overview`: 컬렉션 설명
- `poster_path`: 컬렉션 포스터 경로 또는 URL
- `created_at`: 생성 시각
- `updated_at`: 수정 시각

#### 왜 필요한지

일부 영화와 콘텐츠는 시리즈나 프랜차이즈 단위로 추천될 수 있습니다. 예를 들어 같은 세계관이나 연속 작품을 함께 추천하려면 컬렉션 개념이 도움이 됩니다.

#### 향후 확장 가능성

- 시리즈 정주행 추천을 만들 수 있습니다.
- 프랜차이즈별 페이지를 만들 수 있습니다.
- 후속작, 전작, 관련 작품 추천에 사용할 수 있습니다.

## 4. Content Domain 설계 원칙

- 콘텐츠 메타데이터와 사용자 개인정보를 분리합니다.
- 외부 데이터 ID와 내부 ID를 분리합니다.
- 장르, 국가, 언어, 플랫폼은 문자열이 아니라 표준화된 참조 데이터로 관리합니다.
- 추천 UI에 필요한 최소 정보부터 설계하고, 커뮤니티와 개인화 데이터는 별도 도메인으로 분리합니다.
- 실제 구현 전에는 v1.0에서 꼭 필요한 테이블만 다시 좁힙니다.

## 5. 다음 검토 사항

- TMDB 응답 필드를 `contents`, `people`, `genres` 후보와 어떻게 매핑할지 정합니다.
- `contents`에 모든 타입을 담을지, 장기적으로 타입별 상세 테이블을 분리할지 검토합니다.
- `content_platforms`의 제공 정보 출처와 갱신 주기를 정합니다.
- 다국어 제목과 줄거리를 별도 테이블로 분리할지 검토합니다.
- 추천 로그, Watch Later, 댓글, 좋아요는 Content Domain이 아니라 User/Community Domain에서 별도 설계합니다.
