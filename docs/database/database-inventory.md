# Database Inventory Draft

작성일: 2026-07-01

이 문서는 MyOTT에서 지금까지 설계한 DB 테이블 후보를 한곳에 모아 정리한 인벤토리입니다. 실제 DB 구현, SQL 작성, ERD 작성은 하지 않습니다.

## 1. 목적

- Content, User, AI, Community 도메인에서 나온 테이블 후보를 한눈에 확인합니다.
- v1.0 MVP에 필요한 테이블과 이후로 미룰 테이블을 구분합니다.
- 같은 의미로 사용된 후보명을 정리해 다음 DB 설계 단계의 기준을 만듭니다.

## 2. 버전 기준

| 버전 | 의미 |
| --- | --- |
| v1 | 출시 MVP. 추천, TMDB 기반 콘텐츠, OTT 필터, 로그인, 취향 저장, Watch Later 중심입니다. |
| v2 | 출시 후 확장. 댓글, 좋아요, 추천/비추천, Community Picks, 알림, 종료 예정/신규 공개 예정 중심입니다. |
| v3 | 장기 확장. 만화, 웹툰, 소설, 게임, 음악 등 다른 콘텐츠 타입 확장입니다. |
| Parking Lot | 필요성은 있지만 아직 범위, 정책, 우선순위가 정해지지 않은 후보입니다. |

## 4. Content Domain

| 테이블 후보 | 목적 | MVP 필수 여부 | 출시 버전 | 왜 필요한지 | MVP 제외 사유 | 향후 확장 가능성 |
| --- | --- | --- | --- | --- | --- | --- |
| `contents` | 작품 기본 메타데이터 저장 | 필수 | v1 | 추천 결과와 상세 Layer의 기준 콘텐츠가 필요합니다. 제목, 타입, 줄거리, 포스터, 공개일 같은 핵심 정보를 담습니다. | 해당 없음 | 영화, 드라마, 애니 외 장기 콘텐츠 타입으로 확장할 수 있습니다. |
| `content_details` | 상세 표시용 부가 정보 분리 | 선택 | v1 | 상세 Layer 정보가 커질 경우 `contents`에서 표시용 상세 데이터를 분리할 수 있습니다. | MVP에서는 `contents`에 줄거리와 포스터를 함께 둘 수 있어 필수는 아닙니다. | 타입별 상세 테이블이나 다국어 상세 정보로 확장할 수 있습니다. |
| `external_content_ids` | TMDB 등 외부 ID 매핑 | 선택 | v1 | 외부 API의 콘텐츠 ID와 MyOTT 내부 ID를 안정적으로 연결하기 위해 필요합니다. | 초기에는 `contents`의 외부 출처/ID 컬럼으로 처리할 수 있습니다. | TMDB 외 다른 데이터 공급자를 추가할 때 유용합니다. |
| `content_aliases` | 입력 작품명과 실제 작품 매칭 | 선택 | v1 | 사용자가 입력한 한국어/영문/별칭 제목을 실제 콘텐츠와 연결하는 데 필요합니다. | 초기 검색은 TMDB 검색 결과와 `contents` 제목만으로 시작할 수 있습니다. | 오타 보정, 원제/번역제, 검색 품질 개선으로 확장할 수 있습니다. |
| `genres` | 장르 기준 데이터 저장 | 필수 | v1 | Quick Pick과 추천 결과의 장르 표시가 같은 기준을 사용해야 합니다. | 해당 없음 | 장르 번역, 장르 그룹, AI 태그와 연결할 수 있습니다. |
| `content_genres` | 콘텐츠와 장르 연결 | 필수 | v1 | 하나의 콘텐츠가 여러 장르를 가질 수 있어 연결 테이블이 필요합니다. | 해당 없음 | 장르별 가중치나 대표 장르 표시로 확장할 수 있습니다. |
| `people` | 감독, 배우 등 인물 저장 | 선택 | v1 | 추천 카드와 상세 Layer에 감독과 주요 배우를 표시하기 위해 필요합니다. | MVP에서는 문자열로 먼저 표시할 수 있어 정규화는 선택입니다. | 배우/감독 기반 추천, 필모그래피, 인물 상세로 확장할 수 있습니다. |
| `content_people` | 콘텐츠와 인물의 역할 연결 | 선택 | v1 | 한 콘텐츠에 여러 인물이 참여하고 인물마다 역할이 다르기 때문입니다. | 인물 기반 탐색이 MVP 핵심은 아니므로 초기에는 단순 문자열로 대체할 수 있습니다. | 캐스팅 순서, 역할명, 제작진 필터로 확장할 수 있습니다. |
| `ott_platforms` | OTT 플랫폼 기준 데이터 저장 | 필수 | v1 | 사용자가 OTT를 선택하고 해당 플랫폼 기준으로 추천을 받을 수 있어야 합니다. | 해당 없음 | 로고, 지역별 브랜드명, 플랫폼 상태를 관리할 수 있습니다. |
| `content_platforms` | 콘텐츠와 OTT 제공 정보 연결 | 필수 | v1 | 같은 콘텐츠가 여러 OTT에서 제공될 수 있어 연결 구조가 필요합니다. | 해당 없음 | 제공 방식, 종료 예정일, 지역별 제공 상태로 확장할 수 있습니다. |
| `content_availability` | 콘텐츠 제공 가능 상태 기록 | 선택 | v2 | OTT 종료 예정과 신규 공개 예정 기능을 분리해 관리할 수 있습니다. | v1에서는 `content_platforms`에 기본 제공 상태만 두어도 충분합니다. | 제공 시작/종료 이력과 알림 기능으로 확장할 수 있습니다. |
| `countries` | 국가 기준 데이터 저장 | 선택 | v1 | Quick Pick의 국가 필터와 콘텐츠 제작 국가를 같은 기준으로 맞출 수 있습니다. | 국가 필터가 단순 선택값이면 초기에는 참조 테이블 없이 시작할 수 있습니다. | 국가별 랭킹, 지역별 추천, 다국가 제작 정보를 지원할 수 있습니다. |
| `languages` | 언어 기준 데이터 저장 | 선택 | v1 | 원어, 자막, 더빙 등 언어 정보를 관리할 수 있습니다. | 현재 MVP의 핵심 필터는 아니므로 필수는 아닙니다. | 자막/더빙 필터, 다국어 제목과 줄거리로 확장할 수 있습니다. |
| `collections` | 시리즈, 프랜차이즈 묶음 | 제외 | Parking Lot | 같은 세계관이나 시리즈 작품을 연결할 때 필요합니다. | 추천 MVP에서 컬렉션 탐색은 핵심 흐름이 아닙니다. | 시리즈 이어보기, 프랜차이즈 추천으로 확장할 수 있습니다. |
| `availability_events` | 제공 종료/변경 이벤트 기록 | 제외 | v2 | OTT 종료 예정 알림과 제공 상태 변경 이력을 다루기 위해 필요합니다. | v1에서는 추천 결과 표시가 우선이고 알림/이력은 후순위입니다. | 종료 예정 추천, 알림 큐, 운영 리포트와 연결할 수 있습니다. |
| `release_schedule` | 신규 공개 예정 일정 저장 | 제외 | v2 | 새로 공개될 콘텐츠를 추천하거나 알림으로 보낼 수 있습니다. | v1 추천 흐름에는 예정작 캘린더가 필요하지 않습니다. | 신규 공개 예정 추천, 관심 작품 알림으로 확장할 수 있습니다. |
| `platform_availability` | 장기 콘텐츠 플랫폼 제공 정보 | 제외 | v3 | 웹툰, 게임, 음악처럼 OTT 외 플랫폼 제공 정보를 다루기 위한 후보입니다. | v1의 범위는 OTT 영상 콘텐츠입니다. | 콘텐츠 타입별 플랫폼 제공 모델로 확장할 수 있습니다. |
| `game_platform_availability` | 게임 플랫폼 제공 정보 | 제외 | v3 | 게임 확장 시 콘솔, PC, 클라우드 제공 정보를 분리할 수 있습니다. | 게임은 v1, v2 범위 밖입니다. | 게임 구독 서비스, 플랫폼별 플레이 가능 여부로 확장할 수 있습니다. |

## 5. User and Session Domain

| 테이블 후보 | 목적 | MVP 필수 여부 | 출시 버전 | 왜 필요한지 | MVP 제외 사유 | 향후 확장 가능성 |
| --- | --- | --- | --- | --- | --- | --- |
| `anonymous_sessions` | 비로그인 사용자 세션 식별 | 필수 | v1 | 회원가입 없이도 추천 흐름과 최소 행동 데이터를 이어가기 위해 필요합니다. | 해당 없음 | 로그인 전환, 동의 상태, 세션 보관 정책과 연결할 수 있습니다. |
| `users` | 로그인 사용자 최소 상태 저장 | 필수 | v1 | Supabase Auth와 별도로 MyOTT에서 필요한 온보딩 상태와 서비스 설정을 관리해야 합니다. | 해당 없음 | 공개 프로필, 취향 초기화, 계정 삭제 정책과 연결할 수 있습니다. |
| `user_preferences` | 로그인 사용자 취향 저장 | 필수 | v1 | OTT, 장르, 국가, 분위기, 러닝타임 같은 명시적 선호를 저장해야 개인화가 가능합니다. | 해당 없음 | 행동 기반 취향, 취향 수정, Taste DNA와 연결할 수 있습니다. |
| `user_ott_preferences` | OTT 선호 분리 저장 | 선택 | v1 | OTT 선택이 핵심 필터라 별도 테이블로 분리할 수 있습니다. | MVP에서는 `user_preferences` 안에 preference type으로 담을 수 있습니다. | OTT별 가중치, 구독 상태, 가족 공유 설정으로 확장할 수 있습니다. |
| `user_content_type_preferences` | 영화/드라마/애니 선호 저장 | 선택 | v1 | 콘텐츠 타입 선호를 독립적으로 관리할 수 있습니다. | 초기에는 `user_preferences`로 통합해도 충분합니다. | 타입별 추천 가중치와 홈 화면 개인화로 확장할 수 있습니다. |
| `preference_tags` | Quick Pick 옵션 기준 데이터 | 필수 | v1 | 장르, 국가, 분위기, 러닝타임 같은 선택지를 일관된 태그로 관리해야 합니다. | 해당 없음 | AI 태그, 운영자 추천 태그, 계절/상황 태그로 확장할 수 있습니다. |
| `user_preference_tags` | 사용자와 취향 태그 연결 | 선택 | v1 | 사용자가 반복 선택한 Quick Pick을 취향으로 저장할 수 있습니다. | MVP에서는 추천 요청 로그만으로도 시작할 수 있습니다. | 선호 강도, 최근성, 명시/암시 선호 구분으로 확장할 수 있습니다. |
| `user_preference_profiles` | 누적 취향 프로필 저장 | 제외 | Parking Lot | 사용자 취향을 사람이 읽기 쉬운 프로필로 요약할 수 있습니다. | `user_preferences`와 AI의 `preference_vectors` 책임이 정리된 뒤 도입하는 편이 안전합니다. | Taste DNA, 공유 가능한 취향 카드로 확장할 수 있습니다. |
| `watch_later` | 나중에 볼래 목록 저장 | 필수 | v1 | 로그인 사용자가 추천 결과를 저장하고 다시 방문했을 때 이어볼 수 있어야 합니다. | 해당 없음 | 알림, 취향 반영, 시청 완료 상태로 확장할 수 있습니다. |
| `notification_settings` | 사용자 알림 설정 저장 | 제외 | v2 | 종료 예정, 신규 공개, 관심 작품 업데이트 알림을 사용자별로 제어해야 합니다. | 알림 기능은 v1 추천 MVP 이후에 다룹니다. | 채널별 수신 설정, 알림 빈도, 마케팅 동의와 연결할 수 있습니다. |
| `notification_queue` | 발송 대기 알림 저장 | 제외 | v2 | 사용자에게 보낼 알림을 예약하고 처리 상태를 관리하기 위해 필요합니다. | v1에서는 알림 발송 시스템을 만들지 않습니다. | 이메일, 푸시, 인앱 알림 큐로 확장할 수 있습니다. |
| `user_profiles` | 커뮤니티용 공개 프로필 | 제외 | v2 | 댓글과 Community Picks가 생기면 공개 표시 이름과 아바타가 필요할 수 있습니다. | v1 로그인은 취향 저장 중심이고 공개 커뮤니티 기능이 없습니다. | 프로필 공개 범위, 차단, 신고 정책과 연결할 수 있습니다. |
| `friend_connections` | 친구 관계 저장 | 제외 | Parking Lot | 친구 추천이나 공유 기능을 만들 때 필요합니다. | 친구 기능은 아직 제품 범위와 개인정보 정책이 정해지지 않았습니다. | 친구 기반 추천, 공유 목록, 초대 기능으로 확장할 수 있습니다. |
| `shared_links` | 추천 결과 공유 기록 | 제외 | Parking Lot | 추천 결과나 취향 카드를 외부로 공유할 때 추적할 수 있습니다. | 공유 기능은 MVP 핵심 흐름이 아닙니다. | 만료 링크, 공개 범위, 공유 전환 분석으로 확장할 수 있습니다. |

## 6. AI and Recommendation Domain

| 테이블 후보 | 목적 | MVP 필수 여부 | 출시 버전 | 왜 필요한지 | MVP 제외 사유 | 향후 확장 가능성 |
| --- | --- | --- | --- | --- | --- | --- |
| `recommendation_requests` | 추천 요청 기록 저장 | 필수 | v1 | 사용자가 어떤 입력과 조건으로 추천을 요청했는지 기록해야 결과 재현과 품질 개선이 가능합니다. | 해당 없음 | 모델 버전, 실험 그룹, 요청 품질 분석으로 확장할 수 있습니다. |
| `recommendation_request_inputs` | 입력 작품명 저장 | 선택 | v1 | 사용자가 입력한 좋아하는 작품을 추천 이유와 연결할 수 있습니다. | 초기에는 `recommendation_requests` 안에 함께 저장할 수 있습니다. | 입력 작품 매칭 성공률, 오타 보정, 취향 추론으로 확장할 수 있습니다. |
| `recommendation_request_filters` | Quick Pick과 OTT 필터 저장 | 필수 | v1 | 필터 선택만으로도 추천받을 수 있으므로 요청 필터를 구조화해야 합니다. | 해당 없음 | 필터별 성과 분석, 추천 옵션 개선으로 확장할 수 있습니다. |
| `recommendation_results` | 추천 결과 목록 저장 | 필수 | v1 | 어떤 콘텐츠가 어떤 요청에 반환되었는지 알아야 클릭과 피드백을 연결할 수 있습니다. | 해당 없음 | 순위, 점수, 추천 이유, 모델 버전 비교로 확장할 수 있습니다. |
| `recommendation_impressions` | 화면 노출 카드 기록 | 선택 | v2 | 실제로 사용자 화면에 노출된 추천 결과를 분석할 수 있습니다. | v1에서는 결과 저장과 클릭 기록만으로도 최소 분석이 가능합니다. | CTR 분석, A/B 테스트, 노출 품질 측정으로 확장할 수 있습니다. |
| `recommendation_result_clicks` | 추천 카드 클릭 기록 | 선택 | v1 | 사용자가 실제로 관심을 보인 추천 결과를 파악할 수 있습니다. | 데이터 수집 범위를 줄이려면 v1 후반에 도입해도 됩니다. | 클릭 기반 랭킹 개선, 상세 Layer 성과 분석으로 확장할 수 있습니다. |
| `recommendation_reason_views` | 추천 이유 노출 기록 | 제외 | Parking Lot | 사용자가 추천 이유를 실제로 확인했는지 분석할 수 있습니다. | 현재는 상세 Layer 클릭만으로도 충분하고, 세부 노출 추적은 과합니다. | 설명 가능한 추천 UX 개선에 활용할 수 있습니다. |
| `recommendation_feedback` | 추천/비추천 등 명시적 피드백 저장 | 제외 | v2 | 사용자가 추천 품질을 직접 평가하면 추천 개선에 강한 신호가 됩니다. | 추천/비추천 UI가 v1 MVP에 포함되지 않습니다. | 좋아요, 싫어요, 이미 봄, 관심 없음 피드백으로 확장할 수 있습니다. |
| `preference_vectors` | 비식별 취향 벡터 저장 | 제외 | v2 | 반복 행동과 선호를 AI 추천에 쓰기 쉬운 형태로 요약할 수 있습니다. | v1에서는 명시적 선호와 요청 로그로 충분하며 벡터 정책이 더 필요합니다. | 유사 사용자 추천, 개인화 랭킹, Taste DNA 기반으로 확장할 수 있습니다. |
| `taste_dna` | 사용자에게 보여주는 취향 DNA | 제외 | v2 | 취향 패턴을 이해하기 쉬운 카드나 프로필로 보여줄 수 있습니다. | v1에서는 취향 저장이 우선이고 시각화 기능은 후순위입니다. | 공유 가능한 취향 카드, 온보딩 개선, AI 설명 UX로 확장할 수 있습니다. |
| `content_similarity` | 콘텐츠 간 유사도 저장 | 선택 | v1 | 비슷한 작품 추천을 빠르고 일관되게 제공할 수 있습니다. | 초기 AI 추천 방식이 확정되기 전까지는 필수로 두지 않습니다. | 임베딩, 장르/인물/분위기 기반 유사도 모델로 확장할 수 있습니다. |
| `watch_later_signals` | Watch Later 행동을 AI 신호로 분리 | 제외 | Parking Lot | 저장 행동을 강한 관심 신호로 추천 모델에 반영할 수 있습니다. | `watch_later`에서 파생 가능하므로 별도 저장은 아직 이릅니다. | 선호 강도 계산, 장기 취향 변화 분석으로 확장할 수 있습니다. |
| `page_view_events` | 화면 진입 이벤트 저장 | 제외 | Parking Lot | 첫 방문과 재방문 흐름을 분석할 수 있습니다. | 분석 이벤트는 개인정보 정책과 동의 설계가 더 필요합니다. | 퍼널 분석, 온보딩 개선, 익명 세션 품질 측정으로 확장할 수 있습니다. |
| `content_click_events` | 일반 콘텐츠 클릭 이벤트 저장 | 제외 | Parking Lot | 추천 외 영역의 콘텐츠 클릭을 분석할 수 있습니다. | v1에서는 추천 결과 클릭만 먼저 다루는 것이 단순합니다. | 홈 화면, 검색 결과, 컬렉션 클릭 분석으로 확장할 수 있습니다. |
| `content_detail_views` | 상세 화면 조회 기록 | 제외 | Parking Lot | 상세 Layer 조회를 콘텐츠 관심 신호로 쓸 수 있습니다. | 추천 결과 클릭과 중복될 수 있어 초기에는 별도 테이블이 필요하지 않습니다. | 상세 정보 품질 개선, 콘텐츠별 관심도 분석으로 확장할 수 있습니다. |
