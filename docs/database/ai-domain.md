# AI Recommendation Domain Database Design Draft

MyOTT의 AI 추천 시스템이 필요로 하는 데이터 구조를 정리한 설계 초안입니다. 이 문서는 추천 품질 개선에 필요한 데이터 후보를 정의하기 위한 문서이며, SQL 작성, ERD 작성, Supabase 연결, TMDB 연결은 포함하지 않습니다.

## 1. AI Domain의 역할

MyOTT AI는 사용자의 개인정보를 학습하지 않는다.

MyOTT AI는 사용자의 익명 행동 데이터와 콘텐츠 메타데이터를 활용하여 추천 품질을 향상시킨다. 사용자의 이름, 전화번호, 주소 같은 신원 정보가 아니라, 어떤 추천 조건을 선택했고 어떤 추천 결과에 반응했는지를 비식별 형태로 분석한다.

### 핵심 역할

- 작품 입력값, Quick Pick, OTT 선택을 바탕으로 추천 요청을 해석한다.
- 콘텐츠 메타데이터를 활용해 사용자의 취향과 가까운 작품을 찾는다.
- 추천 결과 노출, 클릭, 저장, 좋아요, 싫어요 같은 반응을 바탕으로 추천 품질을 개선한다.
- 추천 이유를 사용자가 이해할 수 있는 문장으로 생성할 수 있게 한다.
- 개인정보 원본 없이 익명화된 추천 패턴과 통계만 활용한다.

### 설계 원칙

- 개인정보 원본은 AI 학습 데이터에 넣지 않는다.
- 익명 세션과 로그인 사용자 데이터는 목적과 보관 범위를 분리한다.
- 추천 품질 개선에 필요한 행동 데이터도 최소 범위로 저장한다.
- 추천 이유는 설명 가능하고 사용자가 이해할 수 있어야 한다.
- AI Domain은 Content Domain, User Domain, Community Domain과 느슨하게 연결된다.

## 2. AI가 사용하는 데이터

AI 추천은 아래 데이터를 사용할 수 있다. 단, 모든 데이터는 데이터 정책에 따라 익명화, 최소화, 보관 기간 제한을 적용한다.

### 추천 요청

- 사용자가 추천받기 버튼을 눌렀을 때 생성되는 요청 데이터
- 작품 입력, Quick Pick, OTT 선택, 콘텐츠 타입 선택을 포함할 수 있음
- 추천 품질 분석과 추천 결과 재현에 필요

### 추천 결과

- 추천 요청에 대해 어떤 콘텐츠가 어떤 순서로 반환되었는지에 대한 데이터
- 추천 노출, 클릭, 저장, 피드백과 연결됨
- 모델 평가와 품질 개선에 필요

### 추천 클릭

- 사용자가 추천 카드 또는 상세 Layer를 클릭/조회한 기록
- 추천 결과 중 실제 관심을 받은 콘텐츠를 파악하는 데 필요

### Quick Pick

- 장르, 국가, 분위기, 러닝타임 같은 빠른 취향 선택값
- 입력 작품 없이도 추천 의도를 파악하는 데 필요

### 검색

- 사용자가 입력한 작품명 또는 검색어
- 콘텐츠 매칭, 별칭 처리, 추천 맥락 파악에 필요
- 자유 입력이므로 개인정보가 섞일 가능성을 고려해 짧은 보관 기간과 필터링 정책이 필요

### Watch Later

- 사용자가 나중에 보고 싶다고 저장한 콘텐츠
- 명시적 관심 신호로 활용 가능
- 로그인 사용자 기능으로 취급

### 좋아요

- 추천 결과 또는 콘텐츠에 대한 긍정 피드백
- 취향 프로필과 콘텐츠 선호도 개선에 활용 가능

### 싫어요

- 추천 결과 또는 콘텐츠에 대한 부정 피드백
- 반복 추천 방지와 취향 제외 조건에 활용 가능

### 이미 본 작품

- 사용자가 이미 봤다고 표시한 콘텐츠
- 중복 추천을 줄이고, 시청 이력을 기반으로 유사 추천을 개선하는 데 활용 가능

## 3. AI가 저장하는 데이터

아래는 AI Domain에서 저장을 검토할 수 있는 데이터와 테이블 후보입니다.

### recommendation_log

#### 왜 필요한지

추천 요청이 언제, 어떤 조건으로 발생했는지 기록합니다. 추천 품질을 분석하고, 특정 추천 결과가 왜 나왔는지 추적하기 위한 기준 데이터입니다.

#### 저장 후보

- 추천 요청 ID
- 익명 세션 ID 또는 사용자 ID 후보
- 선택한 OTT
- 선택한 콘텐츠 타입
- 입력 작품 후보
- Quick Pick 필터
- 추천 요청 시간
- 추천 방식 또는 모델 버전

#### 주의점

- 입력 작품에 개인정보가 포함될 가능성을 고려합니다.
- 익명 사용자는 짧은 보관 기간을 적용합니다.
- 원본 IP나 민감정보는 저장하지 않습니다.

### recommendation_result

#### 왜 필요한지

추천 요청에 대해 어떤 콘텐츠가 어떤 순서로 반환되었는지 기록합니다. 추천 클릭률, 저장률, 피드백률을 계산하려면 요청과 결과의 연결이 필요합니다.

#### 저장 후보

- 추천 결과 ID
- 추천 요청 ID
- 콘텐츠 ID
- 노출 순서
- 추천 점수
- 추천 이유 요약
- 추천 생성 시각

#### 주의점

- 추천 이유에 개인정보가 포함되지 않도록 합니다.
- 모델 내부 점수와 사용자에게 보여주는 설명을 분리할 수 있습니다.

### recommendation_feedback

#### 왜 필요한지

사용자가 추천 결과에 대해 좋아요, 싫어요, 관심 없음, 이미 봄 같은 반응을 남길 수 있게 합니다. 추천 품질 개선과 반복 추천 방지에 필요합니다.

#### 저장 후보

- 피드백 ID
- 추천 결과 ID
- 콘텐츠 ID
- 사용자 ID 또는 익명 세션 ID 후보
- 피드백 타입
- 피드백 생성 시각

#### 주의점

- 로그인 사용자 피드백은 취향 프로필에 반영할 수 있습니다.
- 익명 피드백은 집계 통계 중심으로 활용합니다.

### preference_vector

#### 왜 필요한지

사용자의 장르, 분위기, 국가, 러닝타임, 콘텐츠 타입 선호를 수치화하거나 태그화해 추천에 활용합니다. 직접적인 개인정보가 아니라 취향 패턴을 표현하는 데이터입니다.

#### 저장 후보

- 벡터 ID
- 사용자 ID 또는 익명 세션 ID 후보
- 장르 선호 값
- 국가 선호 값
- 분위기 선호 값
- 러닝타임 선호 값
- 콘텐츠 타입 선호 값
- 마지막 계산 시각
- 계산 버전

#### 주의점

- 벡터는 설명 가능한 태그 기반으로 시작합니다.
- 민감한 속성을 추론하거나 저장하지 않습니다.
- 계정 삭제 시 사용자와 연결된 벡터는 삭제합니다.

### taste_dna (후보)

#### 왜 필요한지

사용자에게 자신의 취향을 이해하기 쉬운 형태로 보여주는 요약 데이터입니다. 추천 시스템 내부 계산 결과를 사용자 경험으로 연결합니다.

#### 저장 후보

- 취향 DNA ID
- 사용자 ID
- 대표 장르
- 대표 분위기
- 선호 콘텐츠 타입
- 선호 러닝타임
- 설명 문구
- 생성 시각
- 갱신 시각

#### 주의점

- 로그인 사용자 기능으로 우선 설계합니다.
- 사용자가 원하면 삭제하거나 다시 계산할 수 있어야 합니다.
- 민감한 성향이나 개인정보를 추론하지 않습니다.

### content_similarity

#### 왜 필요한지

콘텐츠 간 유사도를 저장해 "이 작품을 좋아했다면 이 작품도 좋아할 가능성이 높다"는 추천을 빠르게 제공할 수 있습니다.

#### 저장 후보

- 유사도 ID
- 기준 콘텐츠 ID
- 유사 콘텐츠 ID
- 유사도 점수
- 유사도 기준
- 계산 버전
- 갱신 시각

#### 주의점

- 사용자 개인정보 없이 콘텐츠 메타데이터 기반으로 계산할 수 있습니다.
- 장르, 출연진, 감독, 줄거리 키워드, 클릭 패턴 등을 분리해 관리할 수 있습니다.

### recommendation_request_filters

#### 왜 필요한지

추천 요청에 사용된 필터를 구조화해 저장합니다. Quick Pick과 OTT 필터를 분석하고 재현하는 데 필요합니다.

#### 저장 후보

- 필터 ID
- 추천 요청 ID
- 필터 타입
- 필터 값
- 생성 시각

#### 주의점

- 필터 값은 표준화된 태그 또는 코드로 저장합니다.
- 자유 입력 텍스트와 구분합니다.

### recommendation_result_clicks

#### 왜 필요한지

추천 결과 중 어떤 카드가 실제로 클릭되었는지 기록합니다. 추천 품질의 핵심 행동 신호입니다.

#### 저장 후보

- 클릭 ID
- 추천 결과 ID
- 콘텐츠 ID
- 익명 세션 ID 또는 사용자 ID 후보
- 클릭 위치
- 클릭 시각

#### 주의점

- 클릭은 관심 신호일 뿐 실제 만족을 의미하지 않을 수 있으므로 피드백과 분리합니다.
- 익명 클릭은 보관 기간을 제한합니다.

### watch_later_signals

#### 왜 필요한지

Watch Later는 사용자가 명시적으로 관심을 표시한 강한 신호입니다. 추천 개선에 활용할 수 있습니다.

#### 저장 후보

- 저장 신호 ID
- 사용자 ID
- 콘텐츠 ID
- 연결된 추천 결과 ID 후보
- 저장 시각

#### 주의점

- Watch Later 원본 데이터는 User Domain에 두고, AI Domain에서는 추천 신호로 참조하는 방향이 좋습니다.
- 사용자가 Watch Later에서 삭제하면 추천 신호 반영 방식도 재검토해야 합니다.

## 4. AI가 절대로 저장하지 않는 데이터

MyOTT AI는 아래 데이터를 저장하거나 학습 데이터로 사용하지 않습니다.

- 이름
- 전화번호
- 주소
- 생년월일
- 주민등록번호
- 원본 IP
- 민감정보
- 건강 정보
- 종교
- 정치 성향
- 성적 지향
- 결제 정보
- 사용자의 연락처 목록
- 자유 입력에 우연히 포함된 개인정보 원문

### 이유

- 추천 품질 개선에 필요하지 않습니다.
- 유출 시 사용자에게 큰 피해가 될 수 있습니다.
- AI 추천은 콘텐츠 취향 패턴만으로도 충분히 개선할 수 있습니다.
- Data Policy의 최소 수집 원칙과 충돌합니다.

## 5. 향후 확장

### 추천 이유 생성

- 추천 결과마다 사용자가 이해할 수 있는 이유를 생성합니다.
- 입력 작품, Quick Pick, 콘텐츠 메타데이터를 바탕으로 설명합니다.
- 개인정보가 포함되지 않도록 문장 생성 범위를 제한합니다.

### 유사 사용자 추천

- 직접적인 사용자 식별자가 아니라 익명화된 취향 벡터의 유사성을 활용합니다.
- 같은 장르, 분위기, 러닝타임을 선호하는 사용자군의 패턴을 추천에 반영할 수 있습니다.
- 소규모 서비스 초기에는 규칙 기반 또는 집계 기반으로 시작합니다.

### 취향 DNA

- 사용자의 선호 장르, 분위기, 콘텐츠 타입을 요약해 보여줍니다.
- 사용자가 자신의 취향을 수정하거나 초기화할 수 있어야 합니다.
- 로그인 사용자 중심 기능으로 둡니다.

### 날씨 추천

- 날씨에 맞는 콘텐츠를 추천할 수 있습니다.
- 정확한 위치 대신 사용자가 직접 선택한 지역 또는 넓은 지역 단위만 사용하는 방향을 우선합니다.
- 위치 데이터는 개인정보 위험이 크므로 별도 동의가 필요합니다.

### OTT 종료 예정 추천

- 사용자가 이용 중인 OTT에서 곧 내려가는 작품을 우선 추천할 수 있습니다.
- Content Domain의 `content_platforms`와 연결됩니다.
- 사용자의 Watch Later와 결합하면 더 유용한 알림을 만들 수 있습니다.

### 신규 공개 추천

- 곧 공개될 작품을 사용자 취향에 맞춰 추천합니다.
- Content Domain의 공개 일정 또는 플랫폼 제공 정보와 연결됩니다.
- 알림 설정은 로그인 사용자 기능으로 관리합니다.

## 6. 테이블 후보 요약

| 테이블 후보 | 필요한 이유 |
| --- | --- |
| `recommendation_log` | 추천 요청 조건과 시간, 모델 버전을 기록해 추천 결과를 추적하기 위해 필요합니다. |
| `recommendation_result` | 추천 요청에 어떤 콘텐츠가 반환되었는지 저장해 클릭률과 피드백을 분석하기 위해 필요합니다. |
| `recommendation_feedback` | 좋아요, 싫어요, 이미 봄 같은 명시적 반응을 추천 개선에 반영하기 위해 필요합니다. |
| `preference_vector` | 사용자의 취향을 비식별 패턴으로 표현해 개인화 추천에 활용하기 위해 필요합니다. |
| `taste_dna` | 취향 패턴을 사용자에게 이해하기 쉬운 형태로 보여주기 위해 필요한 후보입니다. |
| `content_similarity` | 콘텐츠 간 유사도를 저장해 빠르고 일관된 유사 작품 추천을 제공하기 위해 필요합니다. |
| `recommendation_request_filters` | Quick Pick과 OTT 필터를 구조화해 추천 요청을 재현하고 분석하기 위해 필요합니다. |
| `recommendation_result_clicks` | 추천 카드 클릭 데이터를 분석해 실제 관심 콘텐츠를 파악하기 위해 필요합니다. |
| `watch_later_signals` | 사용자가 저장한 작품을 강한 관심 신호로 추천 품질 개선에 활용하기 위해 필요한 후보입니다. |

## 7. 구현 경계

- SQL은 작성하지 않습니다.
- ERD는 그리지 않습니다.
