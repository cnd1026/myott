# MyOTT Data Policy and Privacy Design

이 문서는 MyOTT의 데이터 저장 정책과 개인정보 보호 설계 초안입니다. 실제 개인정보 처리방침, 이용약관, 법률 검토를 대체하지 않으며, 서비스 출시 전 관할 법령과 전문가 검토가 필요합니다.

## 1. 데이터 철학

MyOTT는 "필요한 데이터만 저장한다."라는 원칙을 가진다.

MyOTT의 추천 경험은 사용자의 취향을 이해해야 하지만, 사용자의 신원을 과도하게 알아야 하는 서비스는 아니다. 따라서 추천 품질에 직접 필요한 데이터만 제한적으로 저장하고, 신원 확인이나 추천 개선에 필요하지 않은 개인정보는 처음부터 수집하지 않는다.

### 목표

- 개인정보 최소 수집
- 추천 품질 향상
- 확장 가능한 구조
- GDPR 및 개인정보 보호를 고려한 설계
- 익명 사용자도 추천을 사용할 수 있는 구조
- 로그인 사용자는 명확한 동의와 목적 안에서만 취향을 저장하는 구조

### 설계 원칙

- 목적 제한: 데이터는 수집 목적을 먼저 정하고 그 목적 안에서만 사용한다.
- 최소 수집: 추천, 저장, 커뮤니티 기능에 필요한 데이터만 저장한다.
- 보관 기간 제한: 목적이 끝난 데이터는 삭제하거나 익명 통계로 전환한다.
- 개인정보 보호 기본값: 사용자가 별도로 선택하지 않아도 과도한 추적이 일어나지 않게 한다.
- 투명성: 어떤 데이터가 왜 저장되는지 사용자가 이해할 수 있어야 한다.

## 2. 저장하지 않을 데이터

MyOTT는 아래 데이터를 기본적으로 저장하지 않는다.

| 저장하지 않을 데이터 | 저장하지 않는 이유 |
| --- | --- |
| 이름 | 추천 기능에 실명이 필요하지 않다. |
| 생년월일 | 연령 제한 기능이 확정되기 전까지 추천 품질과 직접 관련이 없다. |
| 전화번호 | 로그인, 추천, Watch Later 기능에 필수 정보가 아니다. |
| 주소 | 콘텐츠 추천과 관계없는 고위험 개인정보다. |
| 주민등록번호 | 서비스 목적상 필요하지 않으며 매우 민감한 식별 정보다. |
| 원본 IP | 보안 로그가 필요하더라도 원본 장기 보관은 피하고, 익명화 또는 단기 보관을 우선한다. |
| 민감한 개인정보 | 건강, 종교, 정치 성향, 성적 지향 등은 콘텐츠 추천에 사용하지 않는다. |
| 결제 정보 | 현재 서비스 범위에 결제 기능이 없으므로 저장하지 않는다. |
| 정확한 위치 정보 | OTT 추천에는 국가/지역 수준의 설정이면 충분하다. |
| 사용자의 연락처 목록 | 소셜 추천 기능이 없으므로 필요하지 않다. |

### 원칙

- 저장하지 않아도 되는 데이터는 UI에서도 입력받지 않는다.
- 사용자가 자유 입력란에 개인정보를 입력할 가능성이 있다면 저장 전 필터링 또는 안내를 검토한다.
- 운영 편의를 이유로 민감정보를 로그에 남기지 않는다.

## 3. 익명으로 저장 가능한 데이터

익명 사용자는 로그인 없이 추천을 받을 수 있다. 다만 익명 데이터는 개인 식별을 목적으로 하지 않으며, 추천 품질 개선과 세션 유지에 필요한 범위로 제한한다.

| 데이터 | 저장 목적 |
| --- | --- |
| `anonymous_session_id` | 로그인 없이 같은 세션 안에서 추천 흐름을 유지하기 위해 사용한다. |
| 선택한 OTT | OTT 필터 기반 추천을 제공하기 위해 사용한다. |
| 선택한 장르 | Quick Pick 기반 추천 결과를 조정하기 위해 사용한다. |
| 선택한 국가 | 국가별 콘텐츠 취향을 추천 조건에 반영하기 위해 사용한다. |
| 선택한 분위기 | 가볍게, 여운 있게, 긴장감 같은 추천 맥락을 반영하기 위해 사용한다. |
| 러닝타임 | 짧은 콘텐츠, 2시간 이하, 긴 작품 선호를 추천에 반영하기 위해 사용한다. |
| 추천 요청 시간 | 추천 로그 분석과 오류 추적, 보관 기간 계산에 사용한다. |
| 추천 결과 ID | 어떤 결과가 생성되었는지 재현하고 품질을 분석하기 위해 사용한다. |
| 추천 카드 클릭 | 사용자가 어떤 추천에 관심을 보였는지 집계하기 위해 사용한다. |
| 검색어 | 작품명 매칭과 추천 요청 품질 개선에 사용한다. |
| Quick Pick 사용 여부 | 입력 기반 추천과 필터 기반 추천의 사용 패턴을 비교하기 위해 사용한다. |

### 익명 데이터 처리 원칙

- 익명 세션은 사용자를 직접 식별하지 않는다.
- 익명 세션 데이터는 짧은 보관 기간을 기본값으로 둔다.
- 로그인 전환 시 익명 데이터를 계정에 연결할지 여부는 사용자에게 명확히 안내한다.
- 익명 데이터는 개인 단위 추적보다 기능 개선용 집계에 우선 사용한다.

## 4. 로그인 사용자만 저장하는 데이터

로그인 사용자는 개인화 기능을 위해 아래 데이터를 저장할 수 있다. 이 데이터는 계정 기능과 연결되므로 사용자가 삭제, 수정, 내보내기할 수 있는 구조를 고려한다.

| 데이터 | 저장 목적 |
| --- | --- |
| 취향 프로필 | 반복 선택, 클릭, 저장 데이터를 바탕으로 추천 품질을 개선한다. |
| Watch Later | 사용자가 나중에 보고 싶은 작품을 다시 찾을 수 있게 한다. |
| 추천 기록 | 이전 추천 요청과 결과를 기반으로 추천 맥락을 이어간다. |
| 댓글 | 커뮤니티 기능에서 사용자 의견을 표시한다. |
| 좋아요 | 작품, 댓글, 추천 결과에 대한 선호를 표시한다. |
| 알림 설정 | OTT 종료 예정, 신규 공개 예정 등 사용자가 선택한 알림만 제공한다. |
| 취향 DNA | 사용자의 장르, 분위기, 콘텐츠 타입 선호를 요약해 개인화 추천에 사용한다. |

### 로그인 데이터 처리 원칙

- 로그인 사용자 데이터는 계정 삭제 시 삭제 가능한 형태로 분리한다.
- 커뮤니티 데이터는 표시 이름과 연결될 수 있으므로 별도 공개 범위 정책이 필요하다.
- 취향 DNA는 설명 가능한 형태로 보여주는 것을 목표로 한다.
- 사용자가 직접 저장한 데이터와 자동 생성된 추천 로그를 구분한다.

## 5. 저장 기간 정책

아래 기간은 제안 수준이며, 실제 출시 전 법률 검토와 운영 정책에 따라 조정한다.

| 데이터 유형 | 제안 저장 기간 | 이유 |
| --- | --- | --- |
| 익명 세션 | 30일 | 짧은 재방문과 추천 흐름 유지에 충분한 기간으로 제한한다. |
| 추천 로그 | 90일 | 추천 품질 분석과 오류 재현에 필요한 최소 기간으로 둔다. |
| 추천 결과 클릭 | 90일 | 추천 성능 분석에 사용하되 장기 개인 추적을 피한다. |
| 검색어 | 30일 | 작품 매칭 개선에 사용하되 자유 입력 위험을 고려해 짧게 보관한다. |
| Watch Later | 사용자가 삭제할 때까지 | 사용자가 직접 저장한 명시적 데이터이므로 사용자의 삭제 요청을 기준으로 한다. |
| 취향 프로필 | 계정 유지 기간 동안 | 개인화 추천의 핵심 데이터이며 계정 삭제 시 삭제한다. |
| 댓글 | 삭제 전까지 | 커뮤니티 맥락을 유지하되 사용자의 삭제 기능을 제공한다. |
| 좋아요 | 취소 또는 계정 삭제 전까지 | 사용자의 명시적 반응이므로 취소 가능해야 한다. |
| 알림 설정 | 사용자가 변경 또는 삭제할 때까지 | 알림 기능 제공에 필요하다. |
| 보안 로그 | 30일에서 180일 | 부정 사용 대응 목적에 맞춰 별도 최소 기간을 정한다. |

## 6. 개인정보 삭제 정책

사용자가 계정을 삭제하면 MyOTT는 계정과 직접 연결된 개인정보를 삭제하거나 익명 통계로만 남긴다.

### 계정 삭제 시 삭제할 데이터

- `users` 계정 정보
- 로그인 제공자 연결 정보
- 취향 프로필
- Watch Later 목록
- 추천 기록 중 사용자 ID와 연결된 원본 기록
- 추천 요청 입력값
- 알림 설정
- 사용자별 좋아요
- 사용자별 추천/비추천 피드백

### 계정 삭제 시 처리 방식이 필요한 데이터

- 댓글:
  - 기본 정책은 사용자가 삭제할 수 있게 한다.
  - 계정 삭제 시 댓글도 함께 삭제하거나, 작성자 정보를 제거한 익명 댓글로 남길지 정책 결정이 필요하다.
- 추천 로그:
  - 사용자 식별자를 제거한 집계 통계만 남길 수 있다.
  - 원본 입력값과 개인 식별 가능 연결은 삭제한다.
- 커뮤니티 집계:
  - 좋아요 수, 추천 피드백 수 같은 집계값은 개인 식별 없이 남길 수 있다.

### 익명 통계로 남길 수 있는 데이터

- 장르별 추천 요청 수
- OTT 필터 사용 비율
- Quick Pick 사용 비율
- 추천 카드 클릭률
- 콘텐츠 타입별 선호 분포

### 삭제 정책 원칙

- 계정 삭제는 되돌릴 수 없는 작업으로 안내한다.
- 삭제 완료 시점과 삭제 대상 범위를 사용자에게 설명한다.
- 백업 데이터는 운영상 필요한 최소 기간 이후 삭제되도록 별도 정책을 둔다.
- 법적 의무로 보관해야 하는 데이터가 생기면 개인정보 처리방침에 명확히 적는다.

## 7. AI 학습 정책

MyOTT AI는 개인정보를 학습하지 않는다.

AI 추천은 개인을 식별하는 정보가 아니라 익명화된 추천 패턴과 사용자 행동 통계를 활용하는 방향으로 설계한다.

### 학습에 사용하지 않는 데이터

- 이름
- 이메일 원문
- 전화번호
- 주소
- 원본 IP
- 민감정보
- 자유 입력에 포함된 개인정보
- 삭제 요청이 들어온 계정의 원본 개인 데이터

### 활용 가능한 데이터

- 익명화된 장르 선호 패턴
- Quick Pick 조합별 추천 클릭률
- 콘텐츠 타입별 추천 반응
- OTT 필터별 추천 반응
- 입력 작품과 추천 결과 간의 비식별 매칭 통계

### AI 설계 원칙

- AI 모델 또는 추천 로직에 개인 식별자를 직접 넣지 않는다.
- 추천 이유는 사용자가 이해할 수 있는 방식으로 설명한다.
- 민감한 속성을 추론하거나 표시하지 않는다.
- 사용자가 원하지 않으면 개인화 추천 기록을 삭제할 수 있어야 한다.
- AI 학습 데이터와 서비스 운영 로그를 분리해 관리한다.

## 8. 향후 고려사항

아래 항목은 출시 전 검토해야 할 개인정보 및 규제 체크리스트다.

- 쿠키 동의
- 개인정보 처리방침
- 이용약관
- 개인정보보호법(PIPA)
- GDPR
- CCPA/CPRA (캘리포니아)
- COPPA (아동 서비스일 경우)
- UK GDPR
- LGPD
- APPI
- PDPA
- Privacy Act
- PIPEDA

### 검토 메모

- GDPR 관점에서는 목적 제한, 데이터 최소화, 보관 기간 제한, 보안, 책임성 원칙을 제품 설계에 반영한다.
- CCPA/CPRA 관점에서는 열람, 삭제, 정정, 판매 또는 공유 거부, 민감정보 제한 권리 대응 여부를 검토한다.
- COPPA 대상 서비스가 될 가능성이 있다면 아동 대상 여부, 부모 동의, 아동 데이터 삭제 정책을 별도로 설계한다.
- 한국 서비스로 운영할 경우 PIPA와 개인정보보호위원회 기준을 우선 검토한다.
- 여러 국가에 서비스할 경우 지역별 개인정보 처리방침과 데이터 이전 정책이 필요할 수 있다.

## 9. Founder Note

우리는 "최소 수집"을 선택한다.

추천 서비스는 사용자의 취향을 이해해야 하지만, 사용자의 모든 것을 알아야 좋은 추천을 할 수 있는 것은 아니다. MyOTT가 만들고 싶은 경험은 사용자가 안심하고 취향을 맡길 수 있는 추천 도구다. 그래서 처음부터 많이 모으고 나중에 줄이는 방식이 아니라, 꼭 필요한 데이터만 묻고 꼭 필요한 기간만 보관하는 방식을 기본값으로 둔다.

최소 수집은 기능을 느리게 만드는 제약이 아니라, 오래 갈 수 있는 제품의 기반이다. 사용자가 신뢰할 수 있는 서비스가 되어야 추천도, 커뮤니티도, 개인화도 지속될 수 있다. MyOTT의 데이터 정책은 추천 품질과 사용자 신뢰를 함께 지키는 방향으로 발전해야 한다.
